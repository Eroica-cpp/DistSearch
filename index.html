{#}about

Campus Search Engine

We build a campus-wide search engine to retrieve and integrate information in SZU, which is supported by Apache Solr and Apache Nutch. We also apply machine learning and data mining skills, especially recommender system techniques to enhance user experience.

{#}features

GUI

Statistics

Whole www.szu.edu.cn crawling

Board: 15, 000, 000+ Chinese words, aprox. 70000 articles

All the rest: 5, 000, 000+ Chinese words, 17024 articles

Solr's index size:325MB

Total time: 25 hours (no optimiz.) only 3 hours (after optimiz.)

Reranking Magic

We record user's click history/contexts, and exploit them to rerank Solr's query results so as to provide more revelent webpages.

Email Reminder

We record users' behaviors, including historical clicks and corrseponding contexts, which help us exploit users' interests. Then recommend emerging webpages (e.g. new board announcements & news) to those users by email.

Two Modules Available

1. CSE Digest: top news from your feed

2. Shocking News: first hand announcement that you urge to know

Problems we meet (Nutch)

Detect and Add New Webpages
We use a scheduled webpage fetcher. If a webpages changes regularly, the fetcher will also adapt to its periodicity.

<property> <name>db.fetch.schedule.class</name> <value>org.apache.nutch.crawl.AdaptiveFetchSchedule</value> </property>

Problems we meet (Solr)

Chinese Character Tokenizing
We employed IKAnalyzer.

Problems we meet (Solr)

Rebuilding Index
We write a script to send all fetched segments from Nutch to Solr to rebuild the index.

Cold Setup :-(

Problem: not users' historical records for CF (Collaborative Filtering)

Solution: Content-based Approach. NLP techniques apply :)

Sparse & High dimensional :(

Feature vectors has reeeeeeeeealy high dimension, over 250, 000 dimension, yet in average less than 200 not zero features in our case.

Large Scale

Problem: slooooooooooow to traverse whole database

Solution: create multi-level index, implemented by Fortran.
